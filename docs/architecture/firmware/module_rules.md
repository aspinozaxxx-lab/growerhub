# Правила модулей firmware

## Слои

- `core` не зависит от прикладных модулей.
- `modules` реализуют сценарии устройства и работают через context, events, services и drivers.
- `services` инкапсулируют инфраструктуру устройства.
- `drivers` инкапсулируют конкретное железо.
- `util` содержит общие технические функции.
- `config` содержит аппаратные и build-настройки.

## Правила

- MQTT-сообщение формируется и разбирается через кодек или явные модели.
- State формируется в одном месте и не собирается вручную в разных модулях.
- Команды приходят через очередь событий и обрабатываются модулем команд.
- Драйверы не знают о бизнес-сценариях.
- Модули не хранят долговременную конфигурацию вне сервиса хранения.
- Изменение формата state, command, ack или events требует обновления архитектурной документации.

## Документ модуля

Документ firmware-модуля создается только при необходимости для устойчивого модуля в `docs/architecture/firmware/modules/<module>.md`.

Структура документа:

- Назначение.
- Публичная поверхность.
- Используемые модули.
- Алгоритм работы.
- Ограничения.

Лимиты:

- Назначение - максимум 500 символов.
- Публичная поверхность - список.
- Используемые модули - список.
- Алгоритм работы - максимум 1000 символов.
- Ограничения - максимум 700 символов.

Если текст превышает лимит, он ужимается до краткого описания.
